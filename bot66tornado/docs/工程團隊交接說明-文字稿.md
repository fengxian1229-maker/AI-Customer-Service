## 一、這份交接要讓工程團隊先理解什麼

這個專案不是單純做一個會回話的聊天機器人，而是要做一個能接住真實客服場景的客服自動化系統。

目前的機器人已經可以處理一批固定、高頻、風險可控的客服問題，例如存款未到帳、提款未收到、無法提款、流水查詢、忘記密碼、查上一筆案件、轉真人客服等。

但目前這一版仍然比較接近「安全受控的流程型機器人」，不是最終想要的「像真人客服一樣理解客戶訊息、自己判斷下一步、自然回覆客戶」的 LLM 型客服。

因此交接的重點不是請工程團隊只維護現有模板，而是要讓他們理解：

1. 目前機器人做到哪裡。
2. 現在是用什麼方式做到這些功能。
3. 現在遇到不同問題時，機器人會怎麼運作。
4. 我們最終希望機器人達到什麼程度。
5. 接下來需要工程團隊接著做什麼，讓它從模板化流程逐步變成更像真人的智能客服。

## 二、目前機器人做到哪裡

目前最新工作版是 `bot66tornado`。

這一版的定位是「乾淨版客服 Bot」，它把舊版可用的客服文案、按鈕流程、後台查詢、Telegram 後台交接方式，重新整理成比較清楚、比較安全的窄路徑架構。

這版不是完全從零開始，而是從舊版 `workspace-autoreply-clean` 裡面搬出可用內容，再重新整理：

- 主選單。
- 存款問題選單。
- 提款問題選單。
- 忘記密碼教學。
- 如何充值教學。
- 如何提款教學。
- 存款未到帳資料收集流程。
- 提款未收到資料收集流程。
- 無法提款 / 流水查詢流程。
- 上一筆案件查詢流程。
- Telegram 後台送案與回覆。
- LiveChat 回覆客戶。
- 真人客服轉接。
- 客戶沉默後的跟進語與結束語。

目前這版已經不是完全靠人工手動處理所有客戶，而是可以在 LiveChat 裡面自動接住一部分標準問題。

但目前它的核心仍是規則式狀態機，而不是完整的 LLM 自主客服。

也就是說：

- 它知道哪些固定問題該走哪條流程。
- 它知道什麼資料齊全後可以送後台。
- 它知道什麼問題不能自己處理，應該轉真人。
- 它知道等待後台時客戶補資料要追加到同一個案件。
- 它知道 Telegram 後台回覆要經過處理後才能回客戶。

但它還不能做到：

- 完全像真人一樣理解任意自由文字。
- 根據客服手冊自動推理所有情境。
- 自然地延續長對話。
- 在不靠大量模板的情況下，自己生成多變、貼近真人的回覆。

## 三、目前是用什麼方式做到這些功能

目前機器人採用的是「狀態機 + 固定流程 + 嚴格安全規則」的方式。

### 1. LiveChat 收客戶訊息

客戶在 LiveChat 輸入訊息、按按鈕、傳截圖，bot 會透過 LiveChat API 輪詢到這些事件。

系統會先判斷：

- 這個聊天室是不是允許處理的平台 group。
- 這個平台對應哪個品牌，例如 JUE999、GNA777、JG7、PAG99、CUM777、CON777、ZAP69。
- 這個案件目前是新案件，還是已經有進行中的 state。
- 是否已經真人接手；如果真人已經接手，bot 不應該再插話。

### 2. BotEngine 把訊息丟進狀態機

客戶訊息進來後，會進入 BotEngine。

BotEngine 不直接回話，而是把訊息交給 state machine 判斷。

狀態機會根據目前 stage 判斷下一步，例如：

- 目前在主選單。
- 目前在存款資料收集。
- 目前在提款資料收集。
- 目前在等待 Telegram 後台。
- 目前在後台已回覆後等待客戶確認。
- 目前已轉真人。
- 目前已 soft parked，也就是暫時結束或停住。

### 3. 狀態機決定回覆和動作

狀態機會產生 responses 和 actions。

responses 是客戶看得到的內容，例如：

- 傳一段文字。
- 傳選單按鈕。
- 傳教學圖片。
- 傳資料要求。
- 傳等待後台文案。
- 傳轉真人文案。

actions 是系統要做的事情，例如：

- 送 Telegram 主卡。
- 追加補充資料到 Telegram 案件。
- 查後台流水。
- 查上一筆案件。
- 轉真人客服。

### 4. CommandRunner 執行外部動作

狀態機只決定「應該做什麼」，真正對外部系統動作的是 CommandRunner。

CommandRunner 會負責：

- 對 LiveChat 發文字。
- 對 LiveChat 發按鈕。
- 對 LiveChat 發圖片。
- 把案件送到 Telegram finance group/topic。
- 把客戶補充資料追加到 Telegram 同一張主卡。
- 呼叫後台查流水。
- 把 Telegram 後台回覆處理後發回 LiveChat。
- 記錄 runtime state。

### 5. Runtime state 記住案件狀態

每個案件都有狀態記錄。

它會記錄：

- chatId。
- threadId。
- 平台。
- 客戶目前在什麼 stage。
- 已收集到哪些資料。
- 是否已送 Telegram 主卡。
- Telegram 主卡 message id。
- 後台最後一次回覆。
- 最後一則客戶訊息。
- 最後一則 bot 訊息。
- 是否已經轉真人。
- 是否已經 soft parked。

這些狀態是機器人能夠避免重複送案、避免插話、避免把新資料開成新案件的關鍵。

## 四、目前有哪些功能

### 功能 1：主選單分流

客戶進入 LiveChat 後，bot 會提供主選單。

主選單目前分成：

- 存款問題。
- 提款問題。
- 上一筆案件。
- 其他問題。

如果客戶不按按鈕，而是直接輸入自由文字，bot 會盡量判斷是否能對應到已知流程。

例如：

- 客戶說存款沒有到，會進存款問題。
- 客戶說提款沒有收到，會進提款問題。
- 客戶說忘記密碼，會進忘記密碼教學。
- 客戶說要真人，會轉真人。

如果自由文字不夠明確，bot 不會一直猜，會先提醒客戶按選單；如果重複不明確，會轉真人。

### 功能 2：存款未到帳

這是目前最重要的流程之一。

當客戶表示存款未到帳時，bot 會要求兩個資料：

1. 用戶名或註冊電話。
2. 付款成功截圖。

這裡有一個很重要的安全規則：

資料沒有收齊以前，bot 不會送 Telegram 後台。

也就是：

- 只有用戶名，不送 TG。
- 只有截圖，不送 TG。
- 用戶名 + 截圖都有，才送 TG。

送 TG 後，bot 會告訴客戶案件已收到、正在確認、資金安全。

### 功能 3：提款未收到

提款未收到流程和存款未到帳類似。

bot 會要求：

1. 用戶名或註冊電話。
2. 提款申請截圖。

資料齊全後才送 Telegram。

如果客戶在提款流程中提到以下問題，bot 不會硬走模板，而是轉真人：

- 提款顯示的帳戶不是自己的。
- 銀行或 Nequi 資料不對。
- 系統要求的資料客戶沒有。
- 身分文件、PPT、passport、外籍身分造成提款問題。
- 提款通道或顯示帳戶異常。

因為這些問題已經不是單純「提款未收到」，而是帳戶/身份/提款資料安全問題，不能讓 bot 自己猜。

### 功能 4：無法提款 / 流水查詢

當客戶說不能提款，bot 會先引導他提供用戶名或註冊電話。

之後 bot 會查後台 rollover / turnover。

如果後台顯示仍有流水未完成，bot 會回覆剩餘流水與說明。

如果後台顯示流水已達標，但客戶仍不能提款，bot 應該轉真人。

這個流程目前不送 Telegram finance group，因為它是後台 API 可以先處理的查詢。

### 功能 5：忘記密碼

客戶說忘記密碼時，bot 會發送教學圖和說明。

如果客戶照著操作後仍然不能登入，會轉真人。

如果客戶只是說 OK、謝謝、了解，bot 不會再一直追問。

### 功能 6：如何充值 / 如何提款

bot 目前有充值教學和提款教學。

如果客戶看完教學後，直接傳付款截圖，bot 會判斷他可能已經完成付款但未到帳，接進存款未到帳流程。

如果客戶看完提款教學後，直接傳提款截圖，bot 會接進提款未收到流程。

這是為了避免客戶明明已經進入實際問題，bot 還一直停在教學。

### 功能 7：上一筆案件查詢

客戶可以選「上一筆案件」，然後提供：

- 用戶名。
- 註冊電話。
- email。

bot 會查目前 runtime state 裡是否有符合的上一筆案件。

如果找到還在等待後台的案件，bot 會告知仍在審核。

如果找到後台已回覆的案件，bot 會回覆上一筆案件的結果。

如果找不到，bot 會讓客戶回選單或轉真人。

### 功能 8：Telegram 後台送案

當存款或提款案件資料收齊後，bot 會送 Telegram 主卡到對應平台的 finance group/topic。

Telegram 主卡會包含案件類型、平台、LiveChat chatId、threadId、客戶資料、已收集欄位與附件。

後台人員必須 reply 這張主卡，bot 才會把後台回覆轉回客戶。

如果後台人員只是直接在 topic 裡發一般訊息，bot 不會轉給客戶。

這是非常重要的安全規則，避免內部訊息誤傳給客戶。

### 功能 9：Telegram 後台回覆轉客戶

後台人員 reply 主卡後，bot 會接收 Telegram 回覆。

回覆不會原封不動丟給客戶，而是會經過 staff reply processor。

目前做法是：

- 如果有 Anthropic API key，嘗試用 LLM 把後台回覆轉成自然、有禮貌的西文客服話術。
- 如果沒有 key 或 LLM 失敗，走 fallback 文案。
- 不允許 LLM 新增原文沒有的金額、reference、URL、付款狀態。
- 不允許 LLM 把「還在查」改成「已完成」。

這代表目前已經有一小部分 LLM 能力，但只用在後台回覆潤飾，不控制案件事實。

### 功能 10：等待後台期間補資料

案件已送 TG 後，如果客戶又補資料，例如：

- 新截圖。
- 補金額。
- 補交易 reference。
- 補電話。
- 補用戶名。

bot 不會開新案件，而是會 append 到同一個 Telegram 主卡。

這是為了讓後台看到同一個案件的完整上下文。

### 功能 11：等待後台期間客戶追問

如果客戶只是問「多久」「還沒好嗎」「什麼時候到」，bot 第一次會回等待說明。

如果客戶反覆追問，bot 會轉真人。

如果客戶明確表示很生氣、沒人處理、一直同樣話，會轉真人。

### 功能 12：客戶沉默跟進

bot 發出客戶可見訊息後，如果客戶 120 秒沒有回覆，會送一次關心跟進語。

如果跟進後客戶再 2 分鐘還是沒有回覆，會送結束語。

如果聊天室已關閉、bot 不是聊天室成員、LiveChat 回傳 inactive 或 unauthorized，會標記案件，避免一直重試拖慢系統。

## 五、目前遇到不同問題時，機器人會怎麼運作

### 情境 1：客戶說「我的充值沒到」

bot 會判斷為存款未到帳。

接著要求：

- 用戶名或註冊電話。
- 付款成功截圖。

收齊後送 Telegram 後台。

### 情境 2：客戶只傳付款截圖

bot 會先記住截圖，然後要求用戶名或註冊電話。

不會只靠截圖送 TG。

### 情境 3：客戶只傳用戶名

bot 會記住用戶名，然後要求付款截圖或提款截圖。

不會只靠用戶名送 TG。

### 情境 4：客戶說「提款沒收到」

bot 會進提款未收到流程。

要求用戶名/電話 + 提款截圖。

資料齊全後送 TG。

### 情境 5：客戶說「不能提款」

bot 會進無法提款流程。

要求用戶名或註冊電話，然後查後台流水。

### 情境 6：後台顯示還有流水

bot 會回覆客戶還有多少 rollover / turnover 需要完成，並說明流水是依系統有效投注計算。

### 情境 7：後台顯示流水已完成但客戶仍不能提款

bot 應轉真人，因為可能是其他限制、帳戶、通道、身份或系統問題。

### 情境 8：客戶說提款帳戶不是自己的

bot 會轉真人。

這類問題不能用一般提款未收到流程處理。

### 情境 9：客戶忘記密碼

bot 會給忘記密碼教學。

如果客戶仍無法登入，轉真人。

### 情境 10：客戶要求真人

bot 會轉真人，不應繼續模板流程。

### 情境 11：客戶說「ok / gracias」

目前 bot 不會把這視為問題已解決。

這是刻意設計，因為客戶只說謝謝，可能只是禮貌回覆，不代表存款或提款已經完成。

只有客戶明確說：

- 已到帳。
- 已收到。
- 已解決。
- 問題好了。

才會 soft parked。

### 情境 12：Telegram 後台直接在群裡講話，但沒有 reply 主卡

bot 不會轉給客戶。

只有 reply 已記錄主卡，才會回客戶。

## 六、目前這一版的本質限制

目前這一版是「安全優先」的流程機器人。

它最大的優點是：

- 不容易亂講付款結果。
- 不會資料沒收齊就送案。
- 不會把 Telegram 內部一般訊息轉給客戶。
- 不會在真人接手後繼續插話。
- 可以把固定高頻問題流程化。

但它最大的缺點是：

- 還是偏模板。
- 自由文字理解有限。
- 很多自然客服語氣仍靠固定文案。
- 需要工程師一直補規則、補關鍵句型。
- 客戶如果用很生活化、很模糊、很長的說法，bot 不一定能自然理解。
- 客戶多輪溝通時，bot 還不像真人一樣能完整理解上下文。

所以目前這版只能算是「打底版本」，不是最終理想版本。

## 七、最期望的機器人要達到什麼程度

我們最終希望的機器人，不是模板機器人。

理想狀態是：

客戶不管怎麼講，機器人都能像真人客服一樣理解意思，判斷現在該做什麼，然後用自然、有溫度、符合客服手冊規則的方式回覆。

### 理想目標 1：能理解客戶自由文字

客戶不一定會照選單講話。

客戶可能會說：

- 我錢扣了但沒進遊戲。
- 我剛提款，銀行沒收到。
- 為什麼一直叫我打流水。
- 我玩了但流水沒有變。
- 你們又叫我等。
- 我不知道要按哪裡。
- 我沒有那個帳號欄位。
- 我很生氣，沒人處理。

理想 bot 應該能理解這些說法背後真正的客服意圖，而不是只靠關鍵字。

### 理想目標 2：能自動判斷下一步動作

理想 bot 不只是回文字，而是要能判斷下一步應該做什麼。

例如：

- 要不要問用戶名。
- 要不要問截圖。
- 要不要送 Telegram。
- 要不要查後台。
- 要不要把資料追加到同一案件。
- 要不要轉真人。
- 要不要回 SOP。
- 要不要安撫客戶。
- 要不要判定客戶已解決。

也就是說，LLM 不能只是客服文案生成器，而要成為「意圖理解 + 下一步策略判斷」的一部分。

### 理想目標 3：不能違背客服手冊規則

這點非常重要。

即使未來變得更像 LLM，也不能變成亂講的聊天機器人。

它必須遵守客服手冊和公司規則，例如：

- 不能自己判斷存款已成功。
- 不能自己承諾提款一定到。
- 不能創造後台沒有提供的金額或狀態。
- 不能跳過必要資料收集。
- 不能把未收齊資料的案件送後台。
- 不能把內部 Telegram 訊息亂轉客戶。
- 不能在真人接手後繼續插話。
- 不能對客戶做超出客服權限的承諾。

### 理想目標 4：越來越像真人客服

目前 bot 回覆還是有模板感。

理想上，bot 應該能做到：

- 回覆更自然。
- 不要每次都同一句。
- 能承接客戶情緒。
- 能理解客戶已經說過什麼，不重複問。
- 能在客戶不耐煩時換方式說明。
- 能在等待後台時給合理安撫，而不是一直重複模板。
- 能用簡短但有人味的方式回覆。

但這個「像真人」不能用犧牲安全換來。

最終方向應該是：

「LLM 負責理解與表達，規則/工具負責事實與動作。」

## 八、目前需求：需要工程團隊接著做什麼

接下來的需求不是單純「再補幾個模板」。

真正需求是把目前這個安全但模板化的流程 bot，逐步升級成「受客服手冊約束的 LLM 智能客服」。

### 需求 1：確認目前正式環境到底跑哪一版

工程團隊接手第一件事，必須先確認正式環境到底跑哪個資料夾、哪個 process、哪個 `.env`。

目前專案內同時存在：

- `bot66tornado`：最新工作版。
- `workspace-autoreply-clean`：上一代來源/舊版。
- `workspace-autoreply-rag`：RAG 實驗版。
- 其他 historical folder。

不能讓工程師自己猜。

### 需求 2：建立乾淨版本管理

目前根目錄 Git 狀態不乾淨，很多檔案沒有正式 commit。

工程團隊應該：

- 建立乾淨 repo。
- 確認 `.env`、runtime、reports、node_modules 不進 repo。
- 把 `bot66tornado` 固定成可交接版本。
- 建立正式 branch / release / tag。
- 把目前文件、測試、啟停方式一起納入版本管理。

### 需求 3：重建可信測試標準

目前核心測試、主路徑 gate、真實回放、人類種子回放有通過。

但 `batch:path-review` 有大量失敗，原因看起來是測試 oracle 沒跟最新規則同步。

工程團隊需要重新定義：

- 哪些行為算正確。
- 哪些行為算錯誤。
- 客戶說謝謝是否應結案。
- 等待後台多久轉真人。
- 哪些場景應該 soft parked。
- 哪些場景必須轉真人。

這些要先定好，否則未來 LLM 升級會沒有可信驗收標準。

### 需求 4：把客服手冊規則變成可執行 guardrails

未來要越來越像 LLM，就不能只把客服手冊放在文件裡。

需要把客服手冊規則轉成可執行規則，例如：

- 必收哪些資料。
- 哪些狀態不能由 LLM 判斷。
- 哪些詞代表高風險要轉真人。
- 哪些回覆不能對客戶說。
- 哪些承諾禁止生成。
- 哪些案件必須走 Telegram。
- 哪些案件必須先查後台。

也就是要做「客服手冊 → policy / guardrails / tests」。

### 需求 5：導入 LLM 意圖判斷，但不能放掉安全邊界

目前多數意圖靠規則和關鍵字。

下一步應該讓 LLM 協助判斷自由文字，例如：

- 這句是在講存款還是提款？
- 是想查進度，還是補資料？
- 是抱怨，還是已解決？
- 是無法提款，還是提款未收到？
- 是登入問題，還是忘記密碼？
- 是客戶提供資料，還是在問新問題？

但 LLM 只能輸出結構化判斷，例如：

```json
{
  "intent": "deposit_missing",
  "confidence": 0.86,
  "next_action": "ask_missing_screenshot",
  "risk": "normal",
  "needs_human": false
}
```

真正是否送 TG、是否查後台、是否結案，仍要由 deterministic code 根據規則決定。

### 需求 6：讓回覆更自然，但事實仍由工具提供

目前文案偏模板。

可以讓 LLM 生成更自然的客服回覆，但它只能使用系統提供的 facts。

例如系統 facts 是：

```json
{
  "case_type": "deposit_missing",
  "missing_fields": ["payment_screenshot"],
  "collected_fields": ["username"],
  "customer_mood": "impatient"
}
```

LLM 可以產生比較自然的回覆：

「我了解您已經提供帳號了，現在只差付款成功截圖。請您從聊天裡的 + 上傳圖片，我們收到後就能幫您送後台確認。」

但 LLM 不可以自己說：

「您的款項已經在處理中，十分鐘內會到。」

除非後台真的有這個事實。

### 需求 7：做 RAG，但不是做 keyword template clone

可以把客服手冊、SOP、歷史真人客服對話放進知識庫。

但 RAG 的用途應該是：

- 幫助理解客戶問題。
- 找到相關 SOP。
- 找到真人客服類似說法。
- 讓回覆更自然。
- 幫 LLM 遵守客服手冊。

RAG 不應該控制：

- 付款狀態。
- 提款狀態。
- 流水金額。
- Telegram routing。
- LiveChat handoff。
- case closure。

### 需求 8：建立「真人感」評估

如果目標是越來越像真人，就不能只測流程有沒有跑通。

還要測：

- 客戶是否覺得被理解。
- 有沒有重複問已提供資料。
- 有沒有冷冰冰。
- 有沒有一直模板化。
- 是否在客戶生氣時轉真人或換說法。
- 是否清楚告知下一步。
- 是否避免過度承諾。

這部分可以用歷史真人客服資料建立 offline evaluation。

### 需求 9：保留目前安全流程，逐步替換模板層

不建議一開始就把整個 state machine 丟掉。

比較安全的升級方式是：

第一階段：保留現有 state machine，加入 LLM intent classifier。  
第二階段：保留 deterministic action gate，讓 LLM 產生自然回覆。  
第三階段：加入 RAG，讓 LLM 查客服手冊與真人話術。  
第四階段：建立人工審核與 replay eval，確認不違規。  
第五階段：逐步減少死模板，提高自由文字處理能力。  

這樣才能從模板化跳離，同時不失控。

---

## 九、給工程團隊的最重要結論

目前這個 bot 已經完成了安全可控的客服流程底座。

它可以處理固定高頻問題，也已經有 LiveChat、Telegram、後台查詢、runtime state、測試、啟停工具。

但它還不是最終理想中的智能客服。

目前它的價值在於：

- 已經把客服流程拆出來。
- 已經定義哪些事情可以自動化。
- 已經建立哪些事情不能讓機器人亂做。
- 已經有一個安全的 state machine 作為底座。

接下來工程團隊真正要做的，是在這個底座上，把模板式流程逐步升級成受規則約束的 LLM 客服。

最終方向不是「寫更多模板」。

最終方向是：

> 讓 LLM 理解客戶、生成自然回覆；讓 deterministic rules 和 tools 控制事實、動作與安全邊界。

也就是：

- LLM 負責像真人。
- 程式規則負責不出事。
- 客服手冊負責不可違背的政策。
- 後台和 Telegram 負責真實事實來源。

9. 第一階段先做哪些工程整理。
