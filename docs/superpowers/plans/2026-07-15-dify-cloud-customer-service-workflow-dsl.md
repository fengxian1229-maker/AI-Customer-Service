# Dify Cloud Customer Service Workflow DSL Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce an importable Dify Cloud Workflow DSL that implements the approved customer-service routing, RAG, SOP slot loop, unified Reply Plan, Final Reply, and safe external-action boundaries.

**Architecture:** Author one portable Dify `workflow` DSL and validate it as a graph artifact rather than executing external actions. A focused pytest suite checks schema, node/edge coverage, branch contracts, prompt safety, and secret hygiene; an import guide lists the model, knowledge-base, and environment bindings required after upload.

**Tech Stack:** Dify Workflow DSL YAML, Dify core node types, Python 3.12, PyYAML 6, pytest.

## Global Constraints

- The app mode is `workflow`; one run processes one customer message.
- Cross-turn state is supplied through Start inputs and returned through End outputs.
- All customer-visible text is generated by one `Final Reply` LLM node.
- Every business branch must produce the same Reply Plan fields.
- Missing SOP slots must end with `WAITING_SUPPLEMENT` and must not reach an HTTP node.
- The DSL contains no real API key, production URL, knowledge-base content, or cross-workspace dataset ID.
- HTTP nodes reference environment variables and must expose failure/timeout paths.
- `HUMAN_ACTIVE` and `CLOSED` bypass automatic business actions.
