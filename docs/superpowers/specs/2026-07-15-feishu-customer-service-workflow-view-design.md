# 飞书 AI 客服工作流视图设计

## 目标

在现有飞书多维表格中重建一套可直接评审的业务工作流视图。工作流使用飞书真实节点和动作配置，但保持总开关关闭，不执行测试、不发送消息、不写入业务记录。

保留现有“会话信息表”和“工单表”，删除当前粗糙工作流后重新构建。

## 视图原则

- 画布按“客户会话主线 + 状态流转”组织。
- 所有分支和动作使用可读业务名称，不使用“分支 1”“分支 2”等名称表达业务语义。
- 问题改写是独立节点，不与意图识别合并；后续 RAG 检索和 SOP 判断统一使用改写后的问题。
- 意图识别只输出一级路由；具体 SOP 类型由 SOP 节点选择 Skill。
- 各业务节点只生成“回复素材 + Reply Plan”，不直接形成最终客户文案；所有客户可见文本统一经过 `Final Reply` 节点。
- 每条路径明确输入、动作、状态变化、结果及异常去向。
- 个案事实不得由 RAG 回答，必须进入 SOP 或人工处理。

## 主流程

1. 新客户消息进入“会话信息表”。
2. 标准化消息，统一文本、附件和渠道信息。
3. **问题改写节点**结合当前消息与必要的会话上下文，将省略、指代或追问改写成可独立理解的问题，并写入“改写问题”字段。
4. 语言识别节点识别客户语言并写入“语言类型”字段，后续回复保持同一语言。
5. 检查会话状态：若为 `HUMAN_ACTIVE`，停止自动回复并保留人工接管。
6. AI 意图识别读取“改写问题”，输出一级路由：`RAG`、`SOP`、`情绪关怀`、`转人工`、`无法识别`。
7. 各分支生成结构化回复素材、允许事实、必说信息、禁说信息和降级文案。
8. 所有客户可见文本汇入 `Final Reply` 节点，统一润色、语言适配和安全校验。
9. `Final Reply` 输出最终回复后进入命令规划、回写和发送节点。
10. 动作失败进入统一异常闭环：记录失败原因、有限重试、重试失败后转人工。

## 问题改写节点

- 输入：客户消息、必要的当前会话上下文和已识别的附件类型。
- 输出：一条不依赖上文也能理解的“改写问题”。
- 保留客户原意、金额、时间、渠道和否定关系，不补造客户未提供的事实。
- 若消息本身已经完整，则只做轻量规范化，不改变业务含义。
- 若上下文不足以消除指代，保留不确定信息并交给“无法识别”或补充资料流程，不自行猜测。
- 改写失败时保留原始标准化消息作为降级输入，并记录失败原因。

## 统一 Final Reply 节点

`Final Reply` 是所有客户可见文本的唯一出口。RAG、SOP、情绪关怀、转人工、无法识别、资料索要、等待后台、TG 回复总结和后台查询结果都必须先生成 Reply Plan，再进入该节点。

### 上游输入

- 原始消息、改写问题、目标回复语言和近期会话上下文。
- 一级路由、具体意图、当前 SOP Skill、工作流阶段和会话状态。
- 已收集槽位、缺失槽位、SOP 动作、RAG 检索结果、后台查询结果或 TG 回复结果。
- 节点回复模板、允许使用的事实、必须表达的信息、禁止表达的信息和确定性降级文案。

### 处理步骤

1. 根据上游 Reply Plan 组织客户回复，而不是重新判断业务事实。
2. 保持客户语言一致，并根据租户语气进行自然润色。
3. 保留资料索要项、等待通知、转人工通知和 TG 回复中的关键事实。
4. 过滤内部标签、TG 标识、后台组织名称和技术实现细节。
5. 禁止编造到账、完成、退款、处理成功或时间承诺。
6. 校验必说项、禁说项、语言、事实边界和置信度。

### 输出与降级

- 校验通过：输出 `final_response_text`，状态为 `accepted`。
- 有可审计警告但文本仍可用：输出文本并记录 `accepted_with_warnings`。
- 模型超时、失败、空输出或不可用：使用上游提供的确定性降级文案。
- 输出随后进入“命令规划”节点，替换待发送文本，再进入回写和发送。

### 画布展示方式

每个会产生客户文本的节点通过连线汇入一个醒目的中央 `Final Reply｜统一润色与安全出口` 节点。纯副作用动作（创建 TG 工单、追加 TG 工单、后台查询）可以并行执行，但其客户确认文案仍必须走 `Final Reply`。图片等无需润色的媒体动作可以标记为豁免，配套文字仍走 `Final Reply`。

## RAG 节点

RAG 节点目前明确挂载以下四个教程：

1. **账户与登录教程**：注册、登录、忘记密码和账户安全操作。
2. **身份认证教程**：认证要求、所需资料和一般审核说明。
3. **存款教程**：存款方式、支持渠道、限额、手续费、一般到账时间和操作方法。
4. **提款教程**：提款方式、限额、手续费、一般审核时间和操作方法。

RAG 流程为：读取改写问题 → 在四个教程中检索 → 置信度判断 → 生成基于命中教程的回复素材与 Reply Plan → `Final Reply` → 命令规划 → 回写并发送。

边界规则：客户个人交易的存款未到账、提款未到账、无法提款或流水限制，不属于 RAG 教程问答，必须进入 SOP 节点。

## SOP 节点与 Skills

SOP 是一级意图节点。进入 SOP 后，由 SOP 节点根据识别结果选择以下 Skill：

每个 Skill 共用以下显式节点链，不将槽位逻辑隐藏在一个动作中：

1. **SOP Skill 选择**：从 `deposit_missing`、`withdrawal_missing`、`withdrawal_blocked_or_rollover` 中选中一个 Skill。
2. **槽位提取**：从当前消息、附件和历史槽位中提取候选资料，并记录来源与置信度。
3. **槽位合并与保护**：合并新旧资料；允许客户明确更正，禁止低置信度结果覆盖可靠资料。
4. **缺失槽位计算**：对照当前 Skill 的必填槽位，输出 `slot_memory` 和 `missing_slots`。
5. **SOP Policy**：根据缺失槽位、工作流阶段、人工状态和既有 TG 工单决定下一动作。
6. **回复素材／Reply Plan**：生成索要资料、等待查询、补充已收到或结果通知所需的结构化素材。
7. **Final Reply**：将素材统一润色为客户可见文本并进行安全校验。
8. **动作与状态**：发送客户回复，并按策略创建 TG 工单、追加 TG 工单、查询后台或等待客户补充。

当资料不完整时，流程为：`槽位提取 → 缺失槽位计算 → 索要缺失资料 Reply Plan → Final Reply → 发送 → WAITING_SUPPLEMENT`。客户补充后回到“问题改写／语言识别”，再进入同一 SOP Skill 和槽位提取节点，形成收集循环。

当资料完整时，资金未到账类流程为：`完整性复核 → 生成 TG 工单摘要 → 创建 TG 工单 → 等待后台 → 客户等待文案 Reply Plan → Final Reply`。收到 TG 处理回复后，执行 `TG 回复解析与关键事实总结 → 结果通知 Reply Plan → Final Reply → 客户回复`。

等待后台期间收到客户补充资料时，执行 `补充槽位提取 → 追加 TG 工单摘要 → telegram.append_to_case → 补充已收到 Reply Plan → Final Reply`，不得重复创建主工单。

### Skill：存款未到账（`deposit_missing`）

- Skill 入参／槽位：

  | 参数 | 类型 | 必填 | 说明 |
  | --- | --- | --- | --- |
  | `phone` | 电话／文本 | 是 | 联系电话或注册手机号，客户明确更正时覆盖旧值。 |
  | `receipt_screenshot` | 附件 | 是 | 存款付款凭证截图；同步为 `deposit_screenshot`。 |
  | `account_or_phone` | 文本 | 否 | 用户名、会员账号或注册手机号，用于辅助查询。 |
  | `customer_name` | 文本 | 否 | 客户姓名或称呼，未提供时不得猜测。 |
  | `amount` | 金额 | 否 | 客户明确提供的存款金额。 |
  | `payment_channel` | 枚举／文本 | 否 | 银行卡、GCASH、USDT 等支付渠道。 |

- 完整条件：`phone` 与 `receipt_screenshot` 均存在且附件通过基础校验。
- 资料不完整：状态设为 `WAITING_SUPPLEMENT`，继续向客户收集缺失资料。
- 缺少存款截图时，可先发送示例图片；图片动作豁免润色，索要资料的文字进入 `Final Reply`。
- 资料完整：生成 TG 工单摘要，创建存款未到账 TG 工单并同步“工单表”，状态设为 `WAITING_BACKEND`。
- TG 返回结果后先解析并总结关键事实，再经 `Final Reply` 通知客户；成功则关闭，失败或异常则转人工。

### Skill：提款未到账（`withdrawal_missing`）

  | 参数 | 类型 | 必填 | 说明 |
  | --- | --- | --- | --- |
  | `phone` | 电话／文本 | 是 | 联系电话或注册手机号，客户明确更正时覆盖旧值。 |
  | `receipt_screenshot` | 附件 | 是 | 提款申请凭证截图；同步为 `withdrawal_screenshot`。 |
  | `account_or_phone` | 文本 | 否 | 用户名、会员账号或注册手机号，用于辅助查询。 |
  | `customer_name` | 文本 | 否 | 客户姓名或称呼，未提供时不得猜测。 |
  | `amount` | 金额 | 否 | 客户明确提供的提款金额。 |
  | `payment_channel` | 枚举／文本 | 否 | 提款渠道或收款方式。 |

- 完整条件：`phone` 与 `receipt_screenshot` 均存在且附件通过基础校验。
- 资料不完整：状态设为 `WAITING_SUPPLEMENT`。
- 资料完整：生成 TG 工单摘要，创建提款未到账 TG 工单并同步“工单表”，状态设为 `WAITING_BACKEND`。
- TG 返回结果后先解析并总结关键事实，再经 `Final Reply` 通知客户；成功则关闭，失败或异常则转人工。

### Skill：无法提款／流水限制（`withdrawal_blocked_or_rollover`）

- Skill 入参／槽位：

  | 参数 | 类型 | 必填 | 说明 |
  | --- | --- | --- | --- |
  | `account_or_phone` | 文本 | 是 | 用户名、会员账号或注册手机号，用于查询限制原因或流水要求。 |

- 完整条件：`account_or_phone` 存在并通过非空校验。
- 资料不完整：状态设为 `WAITING_SUPPLEMENT`。
- 资料完整：执行后台查询，状态设为 `WAITING_BACKEND`。
- 后台结果先转为允许事实和结果 Reply Plan，再经 `Final Reply` 返回限制原因、流水要求或下一步处理建议；无法可靠判断时转人工。

## 其他一级路由

### 情绪关怀

生成同理回复素材和 Reply Plan，经 `Final Reply` 形成克制、自然的客户回复。检测到持续负面情绪、投诉升级或风险信息时转人工。

### 转人工

创建“人工转接”工单、分配负责人并生成转接通知 Reply Plan；通知经过 `Final Reply` 后发送，再将会话状态设为 `HUMAN_ACTIVE`。该状态下自动回复节点不得继续执行。

### 无法识别

生成澄清 Reply Plan，经 `Final Reply` 请求客户补充关键对象、时间、金额、截图或期望结果，然后重新进行一次意图识别；仍无法识别则转人工。

## 画布结构

主画布从上至下展示触发、消息标准化、问题改写、语言识别、人工状态保护、意图识别和一级路由。SOP 分支内再放置一个明确命名的 SOP 节点，其下展开三个 Skill；每个 Skill 显式展示入参、槽位提取、缺失槽位计算、SOP Policy、资料索要循环、TG 工单／后台动作和 TG 回复总结。RAG 节点正文中直接列出四个现有教程。所有会产生客户文本的路径汇入中央 `Final Reply`，再进入命令规划、回写、发送和异常处理区域。

## 业务流程图视图

飞书自动化画布只显示动作类型，无法把业务名称和参数完整展示在节点卡片上。因此最终评审图使用独立的业务流程图视图，不以自动化画布截图作为交付图。

- 每个节点在图中直接显示业务名称、关键输入、输出和状态，不依赖右侧任务配置框。
- 主线直接显示：消息进入 → 标准化 → 问题改写 → 语言识别 → 人工状态保护 → 一级意图识别。
- RAG 区域直接列出账户与登录、身份认证、存款、提款四个教程，并显示置信度与越界转 SOP。
- SOP 区域直接列出三个 Skill 及其必填／可选参数，并显示槽位提取、合并、缺失计算、资料收集循环、TG／后台动作和结果总结。
- 情绪关怀、转人工、无法识别分别显示独立 Reply Plan 与升级规则。
- 所有客户可见路径用清晰连线汇入醒目的 `Final Reply｜统一润色与安全出口`。
- Final Reply 下游显示命令规划、字段回写、发送、有限重试和失败转人工。
- 图使用分区、颜色和状态标签辅助阅读，但颜色不作为唯一语义载体。
- 最终图片插入当前飞书多维表格；现有自动化草稿保留且不启用。

## 安全约束与验收标准

- 删除旧工作流前保留两张现有数据表。
- 不点击“保存并启用”，总开关始终为关闭状态。
- 不运行工作流，不写入测试记录，不发送任何消息。
- 画布中可直接看到一级路由、SOP 节点、三个 Skill 和 RAG 内的四个教程。
- 画布中可直接看到独立的“问题改写”节点，且 RAG 与 SOP 节点使用其输出。
- 画布中可直接看到 `Final Reply` 是所有客户文本的统一出口，包括索要资料、等待通知和 TG 回复总结。
- SOP 区域可直接看到槽位提取、槽位合并、缺失槽位计算、资料收集循环和 TG 补充链路。
- 所有配置节点不显示“未完成设置”。
- 人工接管、补充资料、等待后台、关闭和失败转人工形成完整状态闭环。
- 最终评审图中的业务名称、Skill 入参、状态与 Final Reply 汇流必须直接可见，不要求打开自动化节点详情。
